SwiftCause User Flows v2 — Magic Link Model

Updated post-donation Gift Aid flow with 30-day token expiry

Revision: 16 Apr 2026 (PR #668 in review)

Magic Link Model Overview

Magic links are time-limited, single-use secure URLs that allow donors to complete Gift Aid declarations on their personal devices within 30 days of making a donation. This model decouples donation capture from Gift Aid collection, improving both user experience and completion rates.

Key Features

30-Day Expiry: Tokens are valid for exactly 30 days from the donation timestamp. After expiry, the link returns a 410 Gone response.
O(1) Lookup: Token hash stored as Firestore document ID enables instant lookup without scanning (PR #668).
Pre-Filled Data: Donation amount, date, campaign name, and kiosk location pre-populate the Gift Aid form.
OTP Verification: Email OTP required before form display (security + HMRC compliance, effective 13 Apr 2026).
No Re-use: Each token is single-use. Subsequent access logs and notifies admin (potential fraud signal).

End-to-End User Flows

Happy Path: Donation → Gift Aid

1

Donor taps card at kiosk

User approaches SwiftCause kiosk, selects campaign and donation amount (£5–£10,000). Card tapped via NFC/contactless.

2

Payment processed (Stripe)

Stripe integration charges donor's card. PaymentIntent succeeds. Donation record created in Firestore with status = 'paid'.

3

Magic link token generated

Backend generates 64-char hex token. Token stored as Firestore doc ID for O(1) lookup. QR code rendered from URL: https://swiftcause.co.uk/magic/[token]

4

QR code displayed on kiosk (5-10s)

Kiosk displays QR code with text: "Scan this QR code with your phone to complete your Gift Aid declaration." Auto-return to campaign list after countdown (or manual button).

5

Donor scans QR on personal phone

Donor uses personal mobile phone camera or QR scanner app. QR resolves to magic link. Browser opens to Gift Aid page (can be anytime within 30 days).

6

OTP verification (NEW)

Frontend requests email address. System sends 6-digit OTP to email. Donor enters OTP. OTP valid for 10 minutes. Verified state stored in session.

7

Gift Aid form pre-filled

Form displays with pre-filled fields: Donation Amount (£), Date, Campaign Name, Kiosk Location. Donor enters: Full Name, Address, Postcode, Email (if different).

8

Server-side validation & HMRC check

Postcode validated against HMRC postcode database. Non-UK donations recorded separately (not eligible for claim). Declaration stored in Firestore as 'verified'.

9

Thank you + confirmation email

Frontend displays success message. Confirmation email sent to donor with donation receipt and Gift Aid declaration summary.

Token & Storage Schema

Magic Link Token Format

64-character hexadecimal string generated from crypto.getRandomValues() at donation time.

Example: f4a7c3b2e1d9a6f8c5b2e7a1d4f9c3b6e8a1f4d7c2b5a8e3f6c9d2a5f8b1e4

Stored as: Firestore document ID in /magicLinks/[tokenHash] collection (O(1) lookup via hash index).

Firestore Document Schema

Collection: magicLinks

Document ID: SHA256(token) (tokenHash)

Fields:

  • token (string): Full 64-char token for final verification
  • donationId (string): Reference to donation document
  • donationAmount (number): Donation amount in pence (e.g., 1000 = £10)
  • donationDate (timestamp): Original donation timestamp
  • campaignId (string): Campaign reference
  • campaignName (string): Campaign name for display
  • kioskId (string): Kiosk device ID (for location tracking)
  • createdAt (timestamp): Token generation timestamp
  • expiresAt (timestamp): Expiry time (createdAt + 30 days)
  • status (string): 'pending' | 'completed' | 'expired' | 'error'
  • usageLog (array): Access events with timestamps and IP addresses
  • declarationData (object): Filled-in Gift Aid form fields (null until submission)

Magic Link URL Structure

https://swiftcause.co.uk/magic/[token]

Frontend route: /magic/:token (Next.js dynamic route)

Backend API: GET /api/magic-link/:tokenHash (tokenHash is SHA256(token) for query)

Token passed to frontend in URL fragment (#) for security; never sent to server in logs.

System Architecture

Data Flow

A

Kiosk → Payment (Existing)

Kiosk captures donation via Stripe API. Donation stored in /donations collection.

B

New: Token Generation Cloud Function

Cloud Function triggered on Firestore donation write. Generates token, stores in /magicLinks, returns QR URL. Integrated with kiosk display firmware.

C

Magic Link Page (Frontend)

Next.js page component. Reads token from URL. Fetches pre-filled data via GET /api/magic-link/:tokenHash. Renders form with OTP verification flow.

D

Gift Aid Submission (New Cloud Function)

completeGiftAidFlow() Cloud Function receives form submission. Validates postcode, creates declaration record, triggers HMRC export scheduler. Returns 200 + success JSON.

E

HMRC Export (Existing, Aligned)

Scheduled Cloud Function exports Gift Aid declarations monthly (aligned to April 6 tax year). Generates R68 CSV for HMRC submission. Records batched in /hmrcExports.

Error Handling

Expired Token: Returns 410 Gone. Frontend displays: "This link has expired. Please return to the kiosk and request a new QR code."
Invalid Token: Returns 404. Frontend displays: "Invalid or corrupted link. Please try again."
Postcode Validation Fails: Returns 422 (Unprocessable Entity). Frontend allows retry or manual override (for edge cases).
Token Re-use Detected: Logs security event. Notifies org admin. Does not prevent submission (UX-friendly) but flags for review.

Business Rules & Compliance

HMRC Compliance

1. OTP Required Before Form: Email OTP must be verified before donor sees any Gift Aid form or pre-filled data (effective 13 Apr 2026, closes privacy exposure).
2. Postcode Validation: Main eligibility check for UK Gift Aid. Non-UK postcodes recorded separately and excluded from HMRC claim batches.
3. 30-Day Window: Aligns with HMRC guidance. Tokens expire exactly at midnight UTC on day 30. No extension possible via magic link; org must issue new donation if donor misses deadline.
4. Record Segregation: Gift Aid and non-Gift Aid donations stored in separate Firestore subcollections (/donations/giftAid vs /donations/standard). Export batches exclude non-Gift Aid records.
5. Monthly Batching with April 6 Alignment: R68 export triggered on the 6th of each month (or next business day). Tax year boundary (April 6) automatically resets batch counter.
6. No Donor Contact via Magic Link: System does not send unsolicited follow-up emails via magic link (to avoid Gift Aid spam). Confirmation email sent only after submission.
7. Member Reference (Deferred): Member Ref field required by HMRC but deferred to post-launch (Issue #639, Qamar backlog). For now, field is optional in form; system inserts placeholder UUID.

Security Rules

1. Single-Use Token: Token burned after first successful submission. Subsequent access attempts return 410 Gone after 5 retries (rate limit: 5 lookups per minute per IP).
2. Rate Limiting: Magic link lookup API rate-limited to 5 requests per minute per IP. Brute force protection: blocks IP after 10 failed lookups.
3. Token in URL Fragment (#): Token never logged server-side. Passed in URL fragment to frontend. Frontend hashes token for API calls (double-hashing for security).
4. HTTPS Enforced: All magic links served over HTTPS. Kiosk QR codes redirect from http://swiftcause.co.uk/magic/[token] to https://... to prevent downgrade attacks.
5. Logging & Audit: All token accesses logged (timestamp, IP, user-agent) in usageLog field. Admin dashboard shows suspicious patterns (e.g., repeated failed attempts).

Changes from v1 (Magic Link v1 → v2)

Aspect v1 (Current) v2 (16 Apr PR #668)
Token Lookup Linear scan of /magicLinks collection (O(n) — inefficient for 100k+ records) Direct document ID fetch via tokenHash (O(1) — instant lookup)
Token Storage Stored in array field, searched with Firestore query Stored as document ID (tokenHash) for index-free lookup
OTP Flow Optional, shown after form display Required before form (13 Apr change, security + HMRC requirement)
QR Code Generation Kiosk-side (custom logic) Backend Cloud Function (standardized, tested)
Form Pre-Fill Manual donor entry on kiosk Auto-filled from magic link + Firestore donation record
Expiry Handling Soft delete (archived to old collection) Hard expiry (410 Gone response at token.expiresAt)
Token Re-use Allowed (potential security gap) Single-use, logged on retry, rate-limited
API Endpoint GET /api/magic-links?token=[token] (query param) GET /api/magic-link/:tokenHash (path param, hashed)
Donor Email Capture From QR scan context (unreliable) OTP verification (reliable, secure)
