SwiftCause User Flows

As-Is (Current State) vs To-Be (Target State) Comparison Updated: 2026-04-13

Overview

This document outlines the current user flows (As-Is) and the target optimized flows (To-Be) for the SwiftCause donation platform.

As-Is (Current)

Current implementation focuses on:

  • Donation capture via kiosk/mobile
  • Gift Aid collection post-donation
  • Manual verification of donor details
  • Integration with HMRC Gift Aid service
Status: Functional

To-Be (Target)

Target optimization includes:

  • Optimized card-tap and QR code flows
  • Instant Gift Aid eligibility pre-check
  • Streamlined donor data collection via magic links
  • Enhanced GASDS support for larger campaigns
Status: In Development

Key Differences

The main improvements in the To-Be state focus on reducing friction in the donor journey while maintaining full HMRC compliance for Gift Aid declarations. The magic link technology enables offline-first capture with post-validation on mobile devices.

Donation Capture Flow

As-Is: Current Donation Flow

  1. User approaches kiosk or mobile device
  2. Selects campaign and donation amount
  3. Taps card via NFC/contactless or enters payment details
  4. Payment processed through Stripe
  5. Manual Gift Aid capture if user is willing
  6. Donation recorded in system

Friction Points: Manual data entry, limited time for Gift Aid capture on kiosk

To-Be: Optimized Donation Flow

  1. User approaches device (mobile or kiosk)
  2. Selects campaign and amount (same UX)
  3. Taps card or scans QR code
  4. Payment processed (Stripe, same backend)
  5. QR code displayed for Gift Aid declaration
  6. User scans QR on personal phone → magic link → Gift Aid form
  7. Data collected via mobile web, gift aid stored

Improvements: Self-service on mobile, longer time window, better UX

Magic Link Technology

What is a Magic Link?

A magic link is a unique, time-limited URL (30-day expiry) that allows a donor to securely access their donation record and Gift Aid declaration form without needing to create a username/password. The link is generated at donation time and delivered via QR code.

As-Is: No Magic Link

Gift Aid collection happens immediately after donation, requiring immediate input from donor. Limited by kiosk usability and time constraints.

To-Be: Magic Link Implementation

Donation triggers instant QR code generation. Donor scans on personal device, accesses pre-filled declaration form, completes at their convenience within 30 days.

Technical Details

  • Token Generation: Unique 64-char hex token created at donation time
  • Token Storage: Stored as Firestore document ID for O(1) lookup
  • Expiry: 30-day validity window from donation date
  • Security: HMAC-SHA256 validation, rate limiting on lookup
  • Data: Pre-filled with donation amount, campaign, timestamp

Gift Aid Declaration Flow

As-Is: Immediate Collection

Gift Aid data collected at donation point:

  • Full name (required)
  • Address (required)
  • Postcode (required for validation)
  • Email (optional)
  • Consent checkbox

Limited by time at kiosk and data entry friction.

To-Be: Deferred via Magic Link

Gift Aid data collected via mobile web (30-day window):

  • Pre-filled: Donation amount, date, campaign
  • OTP verification via email (new requirement)
  • Form: Full name, address, postcode, email
  • Consent: Explicit checkbox + audit trail
  • Submission: Server-side validation + HMRC record

Enhanced UX, HMRC compliance, audit trail.

HMRC Validation Rules (Post-13-Apr)

  • OTP Required: Email verification before form display (NEW)
  • Postcode Validation: Main Gift Aid eligibility check
  • Non-UK: Donations from non-UK postcodes are recorded but not eligible for Gift Aid claim
  • Record Segregation: Gift Aid and non-Gift Aid donations tracked separately
  • Monthly Batching: R68 export to HMRC monthly (April 6 tax year alignment)
